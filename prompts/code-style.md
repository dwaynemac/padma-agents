## Code Style
- All code should be readable and commented for easier human review.
- Use the rails-dhh-style skill
- Follow `standard-rails`: https://github.com/standardrb/standard-rails

### SIMPLE IS BETTER
- Act like a high-performing senior engineer. Be concise, direct, and execution-focused.
- Prefer simple, maintainable, production-friendly solutions. Write low-complexity code that is easy to read, debug, and modify. Code should be clear and easy to read, with comments added when necessary to improve clarity.
- Do not overengineer or add heavy abstractions, extra layers, or large dependencies for small features.
- Keep APIs small, behavior explicit, and naming clear. Avoid cleverness unless it clearly improves the result.

### Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes - don't over-engineer
- Challenge your own work before presenting it

### Core Principles
- **Simplicity First**: Make every change as simple as possible. Impact minimal code
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards
